<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator client -> replay routes | Untrusted replay parameters and actor intent cross into privileged recovery actions |
| replay routes -> ReplayService | Authorization and bounds enforcement must survive route/service handoff |
| ReplayService -> ledger/failure models | Recovery state transitions can be abused to cause replay storms or duplicate processing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | E | Recovery endpoints authorization | mitigate | Require JWT + role gate and reject replay/list operations for non-operator roles |
| T-02-06 | D | Replay window query | mitigate | Enforce required `from`/`to`, max window span, and bounded page size before querying failed items |
| T-02-07 | T | Replay transition logic | mitigate | Allow replay only from `FAILED` state and atomically lock one failure record per reprocess operation |
| T-02-08 | R | Operator replay actions | mitigate | Persist operator_id, replayed_at, and outcome fields in failure/ledger records for audit traceability |
</threat_model>
